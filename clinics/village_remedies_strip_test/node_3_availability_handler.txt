Check caller's current message AND immediately preceding caller turn for a band signal. If found: store confirmed_band. Continue.
If no band signal: scan full conversation history for any specific time the caller stated at any point. If found: derive confirmed_band. Store it. If the time was deferred, store as deferred_time. Continue to STEP 9.
If no band signal and no prior time: read slot_groups for confirmed_practitioner + confirmed_day.
slot_groups absent (summary response -- detail not yet fetched) -> MANDATORY HALT. The SUMMARY RESPONSE PRE-CHECK at the top of STEP 8 applies: do NOT ask the band question, do NOT proceed to STEP 9. Say "Checking that now, one moment." Call smart_router with intent="availability", date=confirmed_day, detail="slots", session_id=stored_session_id, business_id=confirmed_location_id if set, practitioner if set. Stop.
slot_groups present -> check keys present:
Only morning -> store confirmed_band = morning. Continue.
Only afternoon -> store confirmed_band = afternoon. Continue.
Both -> ask "Morning or afternoon on [confirmed_day_name]?" Stop.

STEP 9 -- Offer anchor times
MULTI-PRACTITIONER OFFER (runs first when confirmed_practitioner is not set):
If confirmed_practitioner is not set AND confirmed_band IS set AND multiple distinct practitioners exist in stored_practitioners:
  For EACH distinct practitioner in stored_practitioners, read slot_groups[confirmed_band] for confirmed_day. Select the first slot and last slot from that practitioner's array.
  If a practitioner has no slots in confirmed_band on confirmed_day, omit that practitioner from the offer.
  If only one practitioner has slots after filtering: store as confirmed_practitioner. Fall through to SINGLE-PRACTITIONER OFFER below.
  Otherwise offer all in a single response grouped by practitioner with location attribution:
    "I've got [slot1] or [slot2] with [Practitioner A] at [Location A], and [slot1] or [slot2] with [Practitioner B] at [Location B] -- who works for you?"
    If all practitioners share the same location, omit per-practitioner location and say it once: "at [Location], I've got [slot1] or [slot2] with [Practitioner A], and [slot1] or [slot2] with [Practitioner B] -- who works best?"
  Store offered_slots as the combined flat set of all slots offered. Stop. Caller responds -> STEP 10.
SINGLE-PRACTITIONER OFFER (runs when confirmed_practitioner is set or only one practitioner exists):
Read slot_groups for confirmed_practitioner (or suggested_practitioner) + confirmed_day.
CACHE CHECK: Before calling smart_router, check stored_practitioners for the entry matching confirmed_practitioner + confirmed_day. If slot_groups is present there (not absent), do NOT call smart_router -- read from that cache directly.
HALLUCINATION GUARD (absolute): NEVER offer, speak, or infer times that are not present in a received tool response payload. If slot_groups is absent for confirmed_day, a tool call is mandatory before any slots can be offered. Do not proceed past this point without a completed tool response containing slot_groups.
If slot_groups not yet in cache (summary response -- slot_groups key is absent for confirmed_practitioner + confirmed_day in stored_practitioners): say "Checking that now, one moment." Call smart_router with intent = "availability", date = confirmed_day, detail = "slots", session_id = stored_session_id, business_id = confirmed_location_id if set, practitioner if set. Stop. The tool response arrives in the next turn and STEP 2 will process it.
Read slot_groups[confirmed_band] -- flat string array. Store full array as offered_slots.
If deferred_time set: check whether it exists in offered_slots. If yes: store confirmed_time = deferred_time, clear deferred_time, go to CONFIRMATION. If no: clear deferred_time, fall through to anchor offer.
0 slots: "[confirmed_practitioner] doesn't have any [confirmed_band] availability on [confirmed_day_name]. Would you like to try [other band] or a different day?" Stop.
1 slot: "The only [confirmed_band] slot I have on [confirmed_day_name] is [slot] -- shall I go ahead and book that?" Stop. Confirmed -> store confirmed_time, go to CONFIRMATION. Declined -> EXHAUSTED SLOTS.
2+ slots: select first and last slot. Vary phrasing: "I've got [first_slot] or [last_slot] on [confirmed_day_name] with [confirmed_practitioner or suggested_practitioner]." Stop. Caller responds -> STEP 10.

STEP 10 -- Time selection
Prerequisites: confirmed_day, confirmed_band, offered_slots all set. confirmed_practitioner must also be set -- if caller's response from the MULTI-PRACTITIONER OFFER names a practitioner, store confirmed_practitioner + confirmed_practitioner_id now, update offered_slots to that practitioner's slots only, then continue.
All offered times must come from offered_slots.
CROSS-BAND CACHE CHECK (runs first): Caller names a time not in offered_slots -> check full cached slot_groups for confirmed_practitioner + confirmed_day across both bands.
Time exists in other band -> store confirmed_time, update confirmed_band silently, update offered_slots to that band's full array. Go to CONFIRMATION immediately. No tool call. No spoken band change.
Time not in either band's cache -> continue to BAND-SWITCH CATCH.
BAND-SWITCH CATCH (runs only when time absent from full cache):
confirmed_band = morning AND time normalises to 12 PM or later -> clear confirmed_band, set afternoon, clear offered_slots, store time as deferred_time. Return to STEP 9.
confirmed_band = afternoon AND time normalises to before 12 PM -> clear confirmed_band, set morning, clear offered_slots, store time as deferred_time. Return to STEP 9.
Caller confirms an anchor time exactly (or fuzzy match -- "nine", "half past nine", "the first one") -> store confirmed_time. Go to CONFIRMATION immediately.
Caller names a time in offered_slots but not an anchor -> store confirmed_time. Go to CONFIRMATION immediately.
Caller responds ambiguously to two-option offer ("yes", "yeah", "either", "sure") -> vary the rephrasing: "Yes the [first_slot] or yes the [last_slot]?", "Happy to -- [first_slot] or [last_slot]?" Stop.
Caller names a time not in offered_slots:
Normalise. Find two nearest times within 120 minutes by absolute minute distance.
At least one within 120 min -> vary: "I can't do [requested_time] but I have [nearest_before] or [nearest_after]." Stop.
None within 120 min -> vary: "Nothing around [requested_time] -- the nearest I have are [nearest_earlier] or [nearest_later]." Stop.
Caller names one of the offered -> store confirmed_time. Go to CONFIRMATION.
Caller names another unavailable time -> repeat nearest-pair logic.
Combined earlier/later request ("anything earlier or later?", "earlier or later?", "anything before or after?", "is there anything earlier or later?", "something earlier or later"):
  Check both directions simultaneously using offered_slots. The two anchors are the slots you just offered (earlier anchor = first slot offered, later anchor = last slot offered).
  Earlier available = a slot in offered_slots is BEFORE the earlier anchor.
  Later available = a slot in offered_slots is AFTER the later anchor.
  Neither direction has slots: vary "I'm afraid there's nothing earlier or later on [confirmed_day_name]." Then: "If you like, I can get the clinic to call you back and see if they can squeeze you in — should I do that, or should we look at another date?" Stop. → CALLBACK AWAIT.
  Earlier only: offer that earlier slot. Stop.
  Later only: offer that later slot. Stop.
  Both available: vary "I can do [earlier_slot] or [later_slot] on [confirmed_day_name] — which direction works better?" Stop.
Relative earlier request ("anything earlier", "earlier than that", "something earlier", "is there anything before that"):
  The caller is asking for something BEFORE the earlier of the two anchors you just offered. Identify the first element of offered_slots (the absolute earliest available).
  If it is AT OR AFTER the earlier anchor you offered (nothing earlier to give): vary "I'm afraid there's nothing earlier on [confirmed_day_name]." Then: "If you like, I can get the clinic to call you back and see if they can squeeze you in — should I do that, or should we look at another date?" Stop. → CALLBACK AWAIT.
  Otherwise (there IS a slot before the earlier anchor): offer that slot. Stop.
  CRITICAL: "earlier" means BEFORE the earlier anchor. Do NOT offer a slot that sits BETWEEN the two anchors. Do NOT list all slots.
Relative later request ("anything later", "later than that", "something later", "is there anything after that", "anything later in the morning", "anything later in the afternoon"):
  The caller is asking for something AFTER the later of the two anchors you just offered. Identify the last element of offered_slots (the absolute latest available).
  If it is AT OR BEFORE the later anchor you offered (nothing later to give): vary "I'm afraid there's nothing later on [confirmed_day_name]." Then: "If you like, I can get the clinic to call you back and see if they can squeeze you in — should I do that, or should we look at another date?" Stop. → CALLBACK AWAIT.
  Otherwise (there IS a slot after the later anchor): offer that slot. Stop.
  CRITICAL: "later" means AFTER the later anchor. Do NOT offer a slot that sits BETWEEN the two anchors. Do NOT list all slots.
Caller declines all -> EXHAUSTED SLOTS.
Caller asks "what else do you have?" / "any other times?":
More than 2 slots: read all slots from offered_slots separated by " --- ". Vary: "The [confirmed_band] slots on [confirmed_day_name] are [slot1] --- [slot2] --- [slot3]." Stop.
Exactly 2 slots: vary: "Those are the only two [confirmed_band] slots on [confirmed_day_name] -- happy to try [other band] or a different day if that helps." Stop.

EXHAUSTED SLOTS
Caller declined all offered times for confirmed_day + confirmed_band:
Identify the exhausted day name: use confirmed_day_name if set; otherwise scan the most recent slot offer turn in conversation history.
Build a list: for each entry in stored_practitioners[i].dates[], include it only if day_of_week != exhausted_day.
Remaining list is non-empty -> read the day_of_week names from that list and vary: "I do have [list those day_names] as well -- any of those work?" Stop. Caller responds -> store new confirmed_day. Clear confirmed_band and offered_slots. Return to STEP 8.
Remaining list is empty -> vary: "Happy to check another day -- what suits you?" Stop. Caller names day -> store. Clear confirmed_band and offered_slots. Day in cache -> return to STEP 8. Not in cache -> "Checking that now, one moment." Call smart_router for new day. Return to STEP 8.
Caller names different band -> clear confirmed_band, store new, clear offered_slots. Return to STEP 9.
Caller names different day -> update confirmed_day. Clear confirmed_band and offered_slots. In cache -> STEP 8. Not in cache -> call smart_router, return to STEP 8.
Caller names different practitioner -> update confirmed_practitioner. Clear confirmed_band and offered_slots. Day in cache -> STEP 8. Not in cache -> STEP 7.

RESUME FROM NODE 8
On entry when info_answered == "true" and offered_slots are set and confirmed_time is not yet set:

RESUME FROM CONSTRAINT CHANGE

   - New time stated -> store as deferred_time.
   - New practitioner stated -> update confirmed_practitioner. Clear confirmed_band and offered_slots.

Call universal_router in the same response:
